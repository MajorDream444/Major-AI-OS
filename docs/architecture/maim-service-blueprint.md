# MAIM Service Blueprint

Status: foundation blueprint
Owner: Major AI Mindset
Related architecture: `maim-ecosystem-wireframe-v2.md`

This document maps the MAIM ecosystem journey from first discovery to long-term legacy participation.

It is not an implementation plan.

It is the operational blueprint that future product, design, agent, automation, analytics, and technical architecture work should trace back to.

## Mission

The MAIM Service Blueprint defines how people move through the MAIM digital campus:

```txt
majoraimindset.com
Public Sanctuary
        ↓
studio.majoraimindset.com
Member Workshop
        ↓
studio.majoraimindset.com/operator
Protected Back Office
        ↓
Legacy participation
```

Every stage should answer:

- What is the user trying to do?
- What does the human touch provide?
- What does the AI agent assist?
- What data is created?
- What system action happens next?

## Ecosystem Boundary

```txt
MAIM teaches.
HAMAL orchestrates.
AMA agents operate.
Hanzo provides AI infrastructure.
Lux provides sovereignty.
```

This blueprint belongs in MAIM because it describes the education, community, and learning journey.

It should not absorb HAMAL doctrine, AMA operating contracts, Hanzo source code, Lux implementation, or the separate Client Intelligence Workspace Platform.

- Instagram
- YouTube
- podcast appearances
- referrals

```txt
Take the Pressure Test
```

## Public Sanctuary Experience

Domain:

```txt
majoraimindset.com
```

Purpose:

- reduce fear
- build trust
- show the founder
- introduce the ABC foundation
- create one next action

Primary sequence:

```txt
Hero
Founder Welcome
Why MAIM Exists
ABC Foundation
Pressure Test Bridge
Personalized Path Preview
Learning Identity
Meet the Builders
Reserve Your Seat
FAQ
```

The public sanctuary should feel calm, cinematic, and human.

It should not feel like a tool catalog, a generic AI course, or a social media flyer.

## Founder Welcome

The founder welcome is the trust engine.

It should answer:

- why Major built MAIM
- who the room is for
- why people feel overwhelmed
- why the system begins with Awareness, Belief, and Context
- what to do next

The experience should feel like:

```txt
conversation
permission
clarity
invitation
```

Not:

```txt
sales pitch
feature demo
software explainer
```

## Registration and Authentication

Registration starts with a lightweight public form:

- full name
- email
- lane or identity
- session interest
- optional goal

Current safe pipeline:

```txt
Landing Page
  ↓
Google Apps Script
  ↓
Google Sheet
  ↓
Welcome Email dry-run
  ↓
Pipeline event log
```

Future authenticated pipeline:

```txt
Registration
  ↓
Member account
  ↓
Studio profile
  ↓
Learning path
  ↓
Community access
```

Disabled until release-gated:

- live welcome email
- calendar invite
- ManyChat
- CRM write
- paid account provisioning
- certificate issuance

## ABC Studio Onboarding

Domain:

```txt
studio.majoraimindset.com
```

Purpose:

- personalize the learning path
- orient the member
- show progress
- connect live sessions, replays, products, and community

First-session onboarding should capture:

- lane
- current AI comfort level
- biggest pressure point
- desired outcome
- preferred learning rhythm
- consent and communication preferences

Initial Studio promise:

```txt
You are not entering a course catalog.
You are entering a workshop.
```

## Daily Learning Loop

The daily loop is simple:

```txt
One idea
One example
One action
One reflection
```

Each learning unit should produce:

- progress event
- saved action
- optional reflection
- recommended next lesson
- possible prompt or worksheet

The loop must stay beginner-friendly.

Do not turn the Studio into a complex LMS before the learning rhythm is proven.

## Live Webinars and Replay Library

Live knowledge sessions are the heartbeat of MAIM.

Each session should include:

- a clear topic
- one MAIM principle
- one real example
- one action
- replay
- follow-up prompt or worksheet

Replay records should include:

- title
- date
- related ABC letter
- lane fit
- transcript status
- summary
- action item
- related resources

## Pressure Test Assessment

The Pressure Test is the bridge between public curiosity and personalized learning.

It should feel like:

```txt
I understand where I am stuck.
```

Not:

```txt
I took a generic quiz.
```

Outputs:

- pressure score
- recommended entry letter
- suggested lane
- first action
- recommended Studio path

The Pressure Test should eventually route people into:

- Explorer
- Builder
- Architect

## Personalized Learning Paths

Learning paths combine:

- ABC progress
- Pressure Test result
- selected lane
- live session attendance
- saved resources
- community behavior
- purchases
- coaching or cohort participation

Example:

```txt
Start at B - Belief
Identity: Builder
Next action: complete B2
Recommended live room: Context for Creators
Suggested product: Start Here Toolkit
```

Personalization must remain transparent.

People should understand why the system recommended a path.

## Learning Identities

Use identity language, not generic skill-level labels.

Initial identity system:

| Identity | Meaning | Primary Need |
| --- | --- | --- |
| Explorer | First steps | Safety, confidence, orientation |
| Builder | Creating momentum | Practice, resources, repeatable actions |
| Architect | Designing systems | Strategy, automation, leadership |

Avoid language like:

```txt
beginner / intermediate / advanced
```

The preferred language lets people step into a role instead of feeling ranked.

## Community Interactions

Community should support:

- introductions
- action check-ins
- builder wins
- questions
- accountability
- live room follow-up
- peer examples

The community layer should not become noisy by default.

MAIM community should feel like:

```txt
calm momentum
shared practice
visible proof
```

## Product Marketplace

The marketplace should use outcome-named products.

Examples:

| Product Name | Function |
| --- | --- |
| Build Your AI Foundation | ABC workbook / foundation path |
| Start Here Toolkit | beginner templates and prompts |
| Find Your Biggest Bottleneck | Pressure Test productization |
| Context Kit | prompt and context support |
| Builder Lab | applied workshop |

Products should connect to learning stages.

Do not make the marketplace a separate store with no educational context.

## Cohort Progression

Cohorts are deeper guided journeys.

Possible progression:

```txt

  ↓
ABC Studio
  ↓
Pressure Test
  ↓
Builder Lab
  ↓
Cohort
  ↓
Project showcase
  ↓
Legacy path
```

Cohorts should produce:

- milestones
- project artifacts
- attendance records
- reflection records
- certificates or badges when approved
- potential case studies

## Certificates, Badges, and Milestones

Credentials should be earned, not decorative.

Possible milestone types:

- ABC Foundation started
- ABC Foundation completed
- First AI action completed
- First project shipped
- Builder Lab completed
- Cohort completed
- Community contributor
- Legacy portfolio started

Lux may later support sovereignty, provenance, ownership, or credentials.

Do not implement credential minting until Lux architecture is approved.

## Operator Touchpoints

Protected operator surface:

```txt
studio.majoraimindset.com/operator
```

Possible future vault surface:

```txt
vault.majoraimindset.com
```

Operator tasks:

- review registrations
- manage sessions
- publish replays
- update ABC content
- monitor Pressure Test results
- review pipeline events
- manage products
- review community signals
- export reports
- handle errors
- approve automations

The operator console is never the public experience.

## Analytics and Feedback Loops

Measure behavior that improves learning and operations.

Core events:

- visitor arrived
- founder video started
- founder video completed
- Pressure Test started
- Pressure Test completed
- registration submitted
- session attended
- replay watched
- lesson started
- lesson completed
- action submitted
- product viewed
- product purchased
- community check-in posted
- milestone earned
- re-engagement triggered

Analytics should answer:

- Where are people getting stuck?
- Which entry points convert to learning?
- Which lessons create action?
- Which sessions create return visits?
- Which agents need better routing data?

## Automation Events

Automation must remain release-gated.

Current approved dry-run direction:

```txt
Registration
  ↓
Apps Script
  ↓
Google Sheet
  ↓
Pipeline event log
```

Future events:

| Event | System | Gate |
| --- | --- | --- |
| Registration submitted | Apps Script / Google Sheet | Active dry-run foundation |
| Member record created | Airtable / Supabase | Future gate |
| Welcome email drafted | Email system | Dry-run before live |
| Calendar invite drafted | Calendar | Future gate |
| Studio account created | Auth provider | Future gate |
| Learning path assigned | Studio / Agent router | Future gate |
| CRM record updated | CRM | Future gate |
| ManyChat follow-up queued | ManyChat | Future gate |
| Milestone issued | Lux / credential layer | Future gate |

No live automation should be added without a release gate.

## AI Agent Touchpoints

| Stage | User Goal | Human Touch | AI Agent | Data Created |
| --- | --- | --- | --- | --- |
| Discover | Learn about MAIM | Video, post, referral, founder voice | Concierge Agent | Visitor profile |
| Founder Welcome | Decide whether this feels safe | Major's explanation | Concierge Agent | Interest signal |
| Register | Reserve a seat | Email / confirmation copy | Onboarding Agent | Member record |
| ABC Studio | Learn with structure | Live teaching and examples | Learning Coach | Progress record |
| Pressure Test | Find the pressure point | Assessment framing | Evaluator Agent | Skill profile |
| Personalized Path | Know where to start | Recommended next action | Path Router Agent | Learning path |
| Community | Engage with others | Members and moderators | Community Agent | Activity record |
| Products | Purchase useful support | Offer and checkout | Commerce Agent | Order record |
| Cohort | Build with guidance | Major / mentor / facilitator | Cohort Agent | Milestone record |
| Operator Review | Keep the system healthy | Human approval | Operations Agent | Audit event |
| Legacy | Build a durable portfolio | Mentor and founder guidance | Legacy Agent | Achievement record |

Agent behavior must be visible through contracts, event logs, and human approval gates.

Agents should not rely on hidden chat history as operational memory.

## Exit and Re-Engagement Loops

People may pause, leave, or become inactive.

Re-engagement should be respectful and useful.

Possible loops:

- missed live session -> replay reminder
- incomplete Pressure Test -> one reminder
- inactive Studio member -> next small action
- purchased product unused -> setup guide
- cohort graduate -> alumni or legacy invitation
- community contributor -> builder spotlight

Do not over-message.

MAIM should feel like a room people want to return to, not a funnel chasing them.

## Implementation Order

Follow this order:

```txt
1. Service Blueprint
2. Information Architecture
3. Design System
4. Technical Architecture
5. Implementation
```

Do not skip directly from the wireframe into code.

Future implementation should know exactly which journey stage it serves.
